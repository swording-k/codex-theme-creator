# Codex Theme Creator 产品方案

## 一句话定位

用户给 Codex 一句话或一张参考图，Codex 自动创作、安装并验证一套覆盖完整界面的 macOS Codex Desktop 主题。

## 我们解决的问题

现有方案更像“主题播放器”：用户能安装作者做好的预设，但自己创作时往往只能换背景，侧栏、项目、按钮、输入框和已有任务没有形成统一视觉，也很难判断是否真的安装成功。

Codex Theme Creator 把创作过程产品化：

1. AI 理解想法或参考图，提出完整视觉方向。
2. 受控主题协议描述背景、颜色、UI Profile、页面表面和装饰。
3. 确定性脚本完成编译、安全检查、安装、恢复和导出。
4. 兼容性探针分别检查 New Chat 和已有任务。
5. 只有真实运行时与截图都通过，才标记为 verified。

## 目标用户

- 想个性化 Codex，但不会写 CSS 或 Electron 注入代码的普通用户。
- 想制作并分享主题包的设计师、开发者和内容创作者。
- 想为团队、品牌或工作场景统一 Codex 视觉的高级用户。

## 核心使用流程

### 纯文字创作

用户描述氛围、题材和可读性要求，Codex 自动选择 UI Profile、生成背景提示和主题配置，再安装验证。

### 参考图创作

用户提供参考图。Codex 提取色彩、构图、材质和情绪，不把参考图中的 UI 或文字直接烧进背景，最终生成适配 Codex 布局的主题。

### 分享主题

用户确认实机效果后，导出器生成可移植 ZIP，只包含允许发布的主题文件，并清除来源记录中的本机绝对路径。

## MVP 能力

- macOS Codex Desktop。
- 文字想法和本地参考图输入。
- `gt-control`、`glass-studio`、`editorial` 三种完整 UI Profile。
- New Chat 与已有任务分别配置透明度和可读性。
- 主题 schema v2 编译、严格字段校验和路径安全。
- 不允许任意 JavaScript、远程资源和可点击装饰。
- Skill 安装、运行时热升级、原运行时恢复。
- 主题 ZIP 导出和隐私清理。
- 主窗口选择、路由识别和兼容性分级。

## 完成标准

一个主题必须依次满足：

| 状态 | 证明 |
| --- | --- |
| created | `source-theme.json` 编译成功，payload 校验通过 |
| installed | 主题文件进入本机 Dream Skin 主题库 |
| active | 运行时返回的 active theme ID 与目标一致 |
| verified | New Chat 与已有任务均兼容，真实截图可读 |

## 产品原则

- 完整界面优先于单张壁纸。
- 可读性优先于装饰密度。
- 真实截图优先于效果图。
- AI 负责创意，脚本负责安全和可重复性。
- 不主动关闭 Codex，不修改 `.app`、`app.asar` 或应用签名。
- 每次 Codex 更新后重新运行兼容性探针。

## 成功指标

- 首次创作到 `created` 的成功率达到 90%。
- 已安装主题的双路由 `verified` 比例达到 80%。
- 新用户从克隆仓库到安装 Skill 不超过 5 分钟。
- 主题导出包不包含本机绝对路径或未允许文件。
- 首月获得 10 个用户生成主题和 3 个外部贡献者。

## 后续路线

### v0.2

- 交互式主题向导和实时预览。
- 自动对比度评分与截图隐私提醒。
- 更多受控 UI Profile 和装饰槽。

### v0.3

- 主题画廊、版本迁移和兼容性矩阵。
- 一键生成 GitHub Release 素材。
- 用户主题 lint 和贡献检查。

### 长期

- 在可验证的前提下评估 Windows 支持。
- 推动上游采用 schema v2 或拆分为独立 Creator 项目。
- 如果 Codex 提供官方主题 API，优先迁移到官方接口。

## 主要风险

- Codex DOM 结构更新会导致选择器降级：用路由探针和版本化兼容层处理。
- 参考图可能包含版权、肖像或商标：记录来源，公开前人工审核。
- 实机截图可能暴露项目名和对话：导出时提示并要求发布前检查。
- 主题过透明导致已有任务不可读：任务页默认使用更高不透明度。
- 用户误以为已生成等于已生效：始终分开报告四种证据状态。
